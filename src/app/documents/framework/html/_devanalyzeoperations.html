Analyze Operations

 

    

 

There are several basic operations that can be performed in any custom attribute evaluation, as listed below.  Detailed examples of each can be found in the linked chapters.

 

Operation

Description

Average

Returns the average (mean) of all values in the indicated numeric element.

Compute

Returns the result of an equation.

Count

Returns number of items that match a given data condition.

Highest or Lowest

Returns the highest (or lowest) numeric value found within the indicated element.

Maximum or Minimum

The same as highest/lowest but ignores the AssumeZero and always skips over blank/missing elements.  (See Working with Blank or Missing data below.)

Newest or Oldest

Returns the most recent (or most ancient) date value found within the indicated element.

Sum

Returns the total of all values for the indicated numeric element.

Worst

Returns the most severe manner of payment.

 

 

Working with Blank or Missing credit data values

 

AssumeZero option

 

Frequently, dollar amounts (and some numeric amounts) are left blank in the credit bureau responses.  Both the CommonData and MergeData classes include an AssumeZero property which indicates how the CreditSharp framework resolves this data condition.  By default, the AssumeZero property is set to true meaning all blank/missing values will be treated as zero.

 

Consider the following example for how this distinction can influence data analysis.  We have four tradelines being reported, three of which have an explicit value and one that is missing:

DefaultEmptyCalcs option

 

Another point to consider is how attributes that could not logically be calculated should be exhibited.  Consider an attribute that attempts to add up the total available credit could be accessed using a consumer's Home Equity Credit Lines.  The consider the consumer who does not have any HEQ credit lines.  

 

If the reported attribute value defaults to zero it may mistakenly communicate that the consumer has already reached their limit on HEQ credit lines, even though they don't actually have any HEQ credit lines.  Leaving the attribute value blank can help alleviate this possible confusion.

 

The DefaultEmptyCalcs property found in the AnalyzeOptions class controls this presentation, with the default setting being true.

Working with numbers, dates and amounts

 

When performing mathematical evaluations of elements that are numbers, amounts or dates it is important to reference them in their underlying data format.  By default, all data elements are exposed as character strings, so for accurate analysis to be carried out each element must be accessed in it's intrinsic form.  This is facilitated using wrapper functions built into each data class.

 

The wrapper functions are as listed below.  Further discussion of each function can be found in the Derivative Methods topic.

 

Function

Description

AGE

Calculates the number of months between an indicate date and the date the credit report was originally ordered.

 

TradelineSolution monthsOld = (trade, options) => trade.AGE(ComTradelineIds.DateOpen)

AGE2

Calculates the number of days between an indicate date and the date the credit report was originally ordered.

 

TradelineSolution daysOld = (trade, options) => trade.AGE(ComTradelineIds.DateOpen)

AMT

Ensures that an amount data element is presented in numeric form.

 

TradelinePredicate isOverLimit = (trade, options) => trade.AMT(ComTradelineIds.BalanceAmount) >

                                                     trade.AMT(ComTradelineIds.CreditLimitAmount);

DATE

Returns a numeric representation of a date computed as:  ((Date.Year - 1900) * 12) + Date.Month

 

For example, March 17th, 2008 would be displayed as: 1299.

 

This simplifies date math when number of the months between to dates is desired.

 

TradelineSolution monthsOpen = (trade, options) => trade.DATE(ComTradelineIds.DateClosed) -

                                                   trade.DATE(ComTradelineIds.DateOpen)

DATE2

Returns a numeric representation of a date computed as: Date.Year * 10000) + (Date.Month * 100) + Date.Day

 

For example, March 17th, 2008 would be displayed as: 20080317.

 

This simplifies date math when number of the days between to dates is desired.

 

TradelineSolution daysOpen = (trade, options) => trade.DATE2(ComTradelineIds.DateClosed) -

                                                 trade.DATE2(ComTradelineIds.DateOpen)

INV

Reverses the order of character strings.  Really only useful for specifying descending sort criteria in format groups.

 

string reverseName = trade.INV(ComTradelineIds.AccountName)

NBR

Ensures that a numeric data element is indeed presented in numeric form.

 

TradelineSolution timesLate30 = (trade, options) => trade.NBR(ComTradelineIds.TimesLate30)
