XML Merge history display

 

    

 

When bureau information is merged it may sometimes be useful to unwind the merge operation to see which underlying bureau elements were judged to be duplicate items and combined to form the final image.  One example might be when a joint bureau response is merged together, the merge history can show which specific elements from the primary and secondary applicants were combined to create a single merged item.  Likewise, if more than one bureau response is merged together, all of the duplicate items from the original bureau can be included beneath each surviving merged item.

 

Merge history display will be generated for the following (the only ones that are currently merged): Address, Employment, Identification, Inquiry, PublicRecord and Tradeline.

 

The control mechanism for implementing this feature via the MergeHistory property found in the FormatOptionsXml class.  There are two levels of history that can be displayed: Stack and/or Origin.

 

The Stack option is most useful when looking at the merged result from multiple credit bureaus.  For a bit of quick background, the merge operation is carried out in two phases: first, all duplicates are eliminated within each credit bureau; then, all items that remain in each bureau are compared to one another and any duplicates found are combined into the final merged result.

The Stack display option will show the items which remained after the first merge phase, so the deduped items from each bureau that contributed to the each item in the merged output.

 

The Origin option will simply display every duplicate item found in the original bureau data that was combined to form each final merged item. This only references the unmerged bureau data, so there will be no items from the intermediate intra-bureau phase.

The snippet of normalized XML, with both the Stack and Origin options enabled (ie: FormatOptionsXml.MergeHistory = XmlMergeHistory.StackAndOrigin;) that follows illustrates the corresponding file structure that will be generated.  Note that this is a highly abbreviated sample that only shows the structural elements relevant to the merge history options.

 

As can be seen, the <TradelineStackItems> element is the container where stacked merged history can be found while the <TradelineOriginItems> element contains the entire merge record.  This same pattern is followed for the other merged data elements, using <AddressStackItems>, <EmploymentStackItems>, <InquiryStackItems>, etc. as appropriate.
