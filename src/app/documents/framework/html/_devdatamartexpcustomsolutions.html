Processing Custom Solutions

 

Namespace: LuxorTech.CreditSharp.Datamart.Experian

    

 

Experian offers a number of services under the guise of custom solutions which can be used to facilitate decision making or providing inputs to you own custom scoring or data analysis.  The CreditSharp framework includes features to assist in parsing and extracting these custom responses and presenting them in a normalized fashion.

 

Responses from Experian's Instant Prescreen product suite, the Prequalification product suite or any of the data attribute services (Premier Attributes, Model Attribute as well as STAGG, Trend View and Extended View attributes) are each parsed and preprocessed into instances of the normalized ComBureauAttribute class.

 

Attribute Responses

 

Each of the various attribute responses that can be delivered by Experian are automatically pre-processed in two ways by the CreditSharp framework.  

 

First, each attribute name/value pair is extracted into it's own instance of the ExpBureauAttribute class.  This step makes it easier to review the various responses rather than have to manually parse the entire attribute result set (which is often contained in a single field and can be difficult to work with.

 

Once that step is complete, the normalization process can be invoked (using the CommonData.Normalize method).  This transfers each individual native bureau attribute into the commonly defined equivalent.  At this point, the data can be reviewed directly through the CommonData object or extracted into an Xml or Json output.

 

Preapproval Responses (Instant Prescreen or Prequalification)

 

The decisioning responses returned by the Experian Instant Prescreen and Prequalification services are fully customizable and may be different from one subscriber to the next.  In order to preprocess these responses, CreditSharp must be given the necessary knowledge of how the decision details are laid out.  This can be is accomplished by providing parsing instructions to the ExperianData.DecisionSchema property using Java Script Object Notation (JSON) prior to invoking the Parse method.  Here is a hypothetical sample of a JSON input for this purpose:

 

{ "style": "fixed",

  "layout": 

  {

    "msgcode": "65",

    "fields": 

    [

      {"name": "decision", "position": 1, "size": 3},

      {"name": "crossSell", "position": 5, "size": 1},

      {"name": "crossSellId", "position": 6, "size": 2},

      {"name": "notify", "position": 9, "size": 1},

      {"name": "message", "position": 10, "size": 67},

      {"name": "exclude", "position": 78, "size": 1},

      {"name": "excludeCode", "position": 79, "size": 2},

      {"name": "offer", "position": 81, "size": 8}

    ]

  } 

}

 

The above schema would be used to parse the following hypothetical Experian Instant Prescreen into eight distinct fields, each one extracted into an ExpBureauAttribute instance:

 

36101006588ODP Y72 YTHIS IS WHERE ANY MESSAGE TEXT WOULD APPEAR                         N00    1500@

Currently defined decision schemas

 

At this time there is only one defined schema, known as the "fixed" schema.  It is appropriate for use with Experian solutions which return a result within the 361 (Message) segment whose contents comprise a fixed set of fields with constant widths.

 

Note: If this form of schema is not sufficient to define the custom Experian response, feel free to contact Luxor support staff to discuss crafting a definition suitable for your use case.

 

The "fixed" schema structure is as follows:

 

{ "style": "fixed",

  "layout": 

  {

    "msgcode": "nn",

    "fields": 

    [

      {"name": "value1", "position": x, "size": y},

      {"name": "value2", "position": x, "size": y}

    ]

  } 

}

 

 

Schema members:

Element

Content

Description

style

"fixed"

Identifies what type of schema is being provided.

msgcode

two digit numeric value

Provides the key for identifying and locating the appropriate 361 segment.  This corresponds to the Message segment's MessageCode field.

 

It will always be "65" when working with Instant Prescreen output.  Prequalification responses will typical require "57" or "63".

fields

array

One or more entries which provide a label and location for obtaining the service results.

fields: name

any text

Provides the name by which your system knows the particular value.

fields: position

a numeric value

Indicates the offset at which the first character of the value can be found.  This is a one-based figure (ie: the first character in the response would be in position: 1).

fields: size

a numeric value

Indicates the length of the value.
